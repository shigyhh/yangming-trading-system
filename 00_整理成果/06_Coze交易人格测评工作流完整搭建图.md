# Coze 交易人格测评工作流完整搭建图

目标：把“九种交易人格测评”做成一个可实战的 AI 测评系统。

建议不要一口吃成胖子。分三版搭：

1. V1：报告生成器。输入测评结果，直接生成 AI 诊断报告。
2. V2：自动评分器。输入用户答题分数，自动算主人格、副人格、风险等级，再生成报告。
3. V3：随机抽题器。从 720 题库随机抽 27/45/90 道题，用户答完后自动评分并生成报告。

你现在 Coze 画布最适合先做 V1。

---

## 一、V1 工作流：AI 诊断报告生成器

### 1. 工作流结构

```text
开始节点
→ 大模型节点：生成诊断报告
→ 结束节点
```

### 2. 开始节点配置

开始节点只保留一个输入变量。

| 配置项 | 内容 |
|---|---|
| 变量名 | input |
| 类型 | String / 字符串 |
| 是否必填 | 是 |
| 说明 | 用户测评结果文本或 JSON |

不要在第一版里拆很多变量。因为 Coze 测试时最容易出错的地方就是变量层级不一致。

### 3. 大模型节点配置

节点名称：

```text
生成诊断报告
```

模型建议：

```text
豆包 1.8 深度思考
```

输入变量：

```text
input = 开始节点.input
```

如果大模型节点提示“变量值不可为空”，通常就是这里没有绑定开始节点的 `input`。

### 4. 大模型节点系统提示词

```text
你是“阳明心学交易人格 AI 诊断官”。

你的任务是根据用户的九种交易人格测评结果，生成一份股民一看就懂、愿意认可、但不构成投资建议的交易行为诊断报告。

你必须遵守：
1. 不荐股，不预测行情，不承诺收益。
2. 不说“你这个人有问题”，只诊断交易行为模式。
3. 语言要真实、克制、扎心但不羞辱。
4. 必须结合阳明心学的“知行合一、致良知、克己、知止、有定”。
5. 输出必须包含：测评说明、主人格、副人格、当前交易风险、最容易亏钱的场景、最该训练的一项能力、阳明心学提醒、7天行动建议、训练营分层建议。
6. 报告要让用户感觉“被看见”，而不是“被批判”。

合规边界：
本报告只用于交易认知教育、行为觉察和训练分层，不构成任何投资建议、荐股建议、买卖建议或收益承诺。
```

### 5. 大模型节点用户提示词

```text
请根据以下测评结果，生成一份《九种交易人格 AI 诊断报告》。

测评结果：
{{input}}

输出结构必须严格按照以下格式：

《九种交易人格 AI 诊断报告》

一、测评说明
说明本报告只用于交易认知教育和行为觉察，不构成任何投资建议、荐股建议、买卖建议或收益承诺。

二、你的主人格
说明主人格是什么，它代表用户最明显的交易模式。语言要像懂股民的人在说话。

三、你的副人格
说明副人格是什么，它代表压力下容易暴露的问题。

四、当前交易风险
结合风险等级和九人格分数，说明当前最需要警惕的交易风险。

五、你最容易亏钱的3个场景
用具体股民场景描述，不要抽象。

六、你最该训练的一项能力
只给一个最核心训练能力，并说明为什么。

七、阳明心学提醒
结合“知行合一、致良知、克己、知止、有定”中的一个或多个概念，给一句有力量但不玄的提醒。

八、7天行动建议
每天一个动作，必须具体、可执行、低门槛。

九、训练营分层建议
根据主人格推荐训练营：
- 冲动型 / 焦虑型 / 从众型：基础觉察营
- 扛单型 / 偏执型 / 赌徒型：风险修正营
- 拖延型 / 完美主义型：执行强化营
- 平衡型：稳定进阶营

十、结语
结尾要温和承接：你不是不适合交易，而是还没有看见自己最容易重复失控的地方。先修正一个动作，再谈建立系统。

语言风格：
- 真实、克制、扎心但不羞辱。
- 多用“你在某种场景下容易……”而不是“你就是……”。
- 不要出现稳赚、暴富、保证收益、推荐股票、买入卖出点位等表达。
```

### 6. 结束节点配置

结束节点输出：

| 输出变量 | 取值 |
|---|---|
| output | 生成诊断报告.output |

注意：不要选 `开始.input`。要选大模型节点的输出。

### 7. V1 测试输入

右侧“试运行输入”如果开启 JSON 模式，必须这样写：

```json
{
  "input": "用户昵称：测试用户\n测评版本：45题标准版\n主人格：冲动型\n副人格：焦虑型\n风险等级：中高风险\n九人格分数：冲动型82，扛单型58，完美主义型46，偏执型52，焦虑型76，从众型63，赌徒型49，拖延型55，平衡型41\n高分子维度：踏空补偿、频繁看盘\n来源渠道：直播间"
}
```

错误写法：

```json
{
  "nickname": "测试用户",
  "main_type": "冲动型"
}
```

错误原因：开始节点只有一个变量叫 `input`，所以外层必须有 `"input": "..."`。

## 二、V2 工作流：自动评分 + 生成报告

V1 跑通以后再做 V2。

### 1. V2 工作流结构

```text
开始节点
→ 代码节点：自动评分
→ 大模型节点：生成诊断报告
→ 结束节点
```

### 2. 开始节点配置

开始节点输入仍然只保留一个变量：

| 变量名 | 类型 | 说明 |
|---|---|---|
| input | String | 用户答题结果 JSON |

### 3. V2 测试输入格式

```json
{
  "input": "{\"test_version\":\"45题标准版\",\"answers\":[{\"personality\":\"冲动型\",\"sub_dimension\":\"追涨冲动\",\"score\":5},{\"personality\":\"冲动型\",\"sub_dimension\":\"踏空补偿\",\"score\":4},{\"personality\":\"焦虑型\",\"sub_dimension\":\"频繁看盘\",\"score\":5}]}"
}
```

注意：这里为了让 Coze 的 `input` 仍然是字符串，里面的 JSON 要作为字符串传入。

如果 Coze 支持 Object 类型，也可以把开始节点改成 Object，但第一阶段不建议。

### 4. 代码节点输入

代码节点名称：

```text
自动评分
```

输入：

```text
input = 开始.input
```

### 5. 代码节点逻辑

代码节点要做 6 件事：

1. 解析用户答题结果。
2. 按人格类型汇总分数。
3. 按子维度汇总分数。
4. 换算百分制。
5. 判定主人格、副人格、风险等级。
6. 输出一个结构化结果 JSON，交给大模型节点。

### 6. 代码节点伪代码

```javascript
const data = JSON.parse(input);
const answers = data.answers || [];

const riskTypes = ["冲动型","扛单型","完美主义型","偏执型","焦虑型","从众型","赌徒型","拖延型"];
const allTypes = [...riskTypes, "平衡型"];

const scores = {};
const counts = {};
const subScores = {};
const subCounts = {};

for (const type of allTypes) {
  scores[type] = 0;
  counts[type] = 0;
}

for (const item of answers) {
  const type = item.personality;
  const sub = item.sub_dimension;
  const score = Number(item.score || 0);
  if (!scores[type]) scores[type] = 0;
  if (!counts[type]) counts[type] = 0;
  scores[type] += score;
  counts[type] += 1;

  const key = `${type}-${sub}`;
  if (!subScores[key]) subScores[key] = 0;
  if (!subCounts[key]) subCounts[key] = 0;
  subScores[key] += score;
  subCounts[key] += 1;
}

const percentScores = {};
for (const type of allTypes) {
  const maxScore = (counts[type] || 0) * 5;
  percentScores[type] = maxScore > 0 ? Math.round(scores[type] / maxScore * 100) : 0;
}

const sortedRisk = riskTypes
  .map(type => ({ type, score: percentScores[type] || 0 }))
  .sort((a, b) => b.score - a.score);

let main_type = sortedRisk[0]?.type || "";
let sub_type = sortedRisk[1]?.type || "";

if ((percentScores["平衡型"] || 0) >= 84 && sortedRisk.every(x => x.score <= 60)) {
  main_type = "平衡型";
  sub_type = "副人格不明显";
}

let risk_level = "中等风险";
const mainScore = main_type === "平衡型" ? percentScores["平衡型"] : sortedRisk[0].score;
const highRiskCount = sortedRisk.filter(x => x.score >= 80).length;
const balanceScore = percentScores["平衡型"] || 0;

if (mainScore >= 88 || highRiskCount >= 2 || balanceScore <= 40) {
  risk_level = "高风险";
} else if ((mainScore >= 72 && mainScore <= 87) || (balanceScore >= 41 && balanceScore <= 56)) {
  risk_level = "中高风险";
} else if (mainScore >= 56 && mainScore <= 71 && balanceScore >= 57 && balanceScore <= 72) {
  risk_level = "中等风险";
} else if (sortedRisk.every(x => x.score <= 52) && balanceScore >= 72) {
  risk_level = "相对稳定";
}

const top_sub_dimensions = Object.entries(subScores)
  .map(([key, score]) => ({
    name: key,
    score,
    avg: Math.round(score / (subCounts[key] || 1) * 20)
  }))
  .sort((a, b) => b.avg - a.avg)
  .slice(0, 3);

return {
  test_version: data.test_version || "",
  scores: percentScores,
  main_type,
  sub_type,
  risk_level,
  top_sub_dimensions
};
```

不同平台的代码节点返回格式可能不同。如果 Coze 要求返回对象，就返回对象；如果要求赋值到变量，就把这个对象赋给 `output`。

### 7. 大模型节点输入

V2 的大模型节点不再吃开始节点的 `input`，而是吃代码节点输出：

```text
input = 自动评分.output
```

用户提示词改成：

```text
请根据以下自动评分结果，生成一份《九种交易人格 AI 诊断报告》：

{{input}}

后续输出结构同 V1。
```

## 三、V3 工作流：随机抽题 + 自动评分 + 生成报告

这个版本最完整，但不建议一开始就做。

### 1. V3 工作流结构

```text
开始节点
→ 代码节点：随机抽题
→ 结束节点：返回题目列表
```

这是“发题工作流”。

用户答完后，再调用第二个工作流：

```text
开始节点
→ 代码节点：自动评分
→ 大模型节点：生成诊断报告
→ 结束节点
```

为什么拆成两个工作流：

1. 抽题是一次动作。
2. 答题是用户在外部页面完成。
3. 评分和报告是提交后动作。

如果全塞在一个 Coze 聊天工作流里，用户体验会很累。

### 2. 随机抽题输入

```json
{
  "input": "{\"test_version\":\"45题标准版\",\"user_id\":\"u001\"}"
}
```

### 3. 抽题规则

| 版本 | 每种人格抽题 |
|---|---|
| 27题 | 每类 3 道 |
| 45题 | 每类 5 道 |
| 90题 | 每类 10 道 |

### 4. 抽题逻辑

```text
从720题库中：
1. 按人格分组
2. 每组抽固定数量
3. 每组尽量覆盖4个子维度
4. 同一用户优先避开历史答过的题
5. 最后打散题目顺序
```

### 5. V3 的现实建议

Coze 可以做抽题，但如果题库有 720 道，建议最终还是放到网页/数据库里管理。

更稳的正式产品结构：

```text
网页负责：抽题、答题、计分、存数据
Coze负责：生成报告、解释结果、私域陪跑
```

## 四、你现在应该先搭哪一个

你当前画布应该先搭 V1。

不要现在就搭 V2/V3。原因：

1. V1 能最快验证 AI 报告质量。
2. V1 变量最少，不容易卡在 Coze 配置。
3. V1 跑通后，V2 只是多一个代码节点。
4. V2 跑通后，V3 才有意义。

你的当前目标：

```text
输入测试 JSON → 生成完整报告
```

只要这一步成功，就说明 Coze 的“诊断官大脑”已经成型。

## 五、最常见报错

### 1. 变量值不可为空

原因：

```text
开始节点变量叫 input，但测试 JSON 没有 input 字段。
```

解决：

```json
{
  "input": "这里放测评结果"
}
```

### 2. 输出为空

检查：

1. 大模型节点是否有用户提示词。
2. 用户提示词是否引用 `{{input}}`。
3. 大模型节点输入是否绑定了开始节点 `input`。
4. 结束节点是否选择了大模型节点的输出。

### 3. 报告只是复述分数

原因：提示词不够明确。

解决：要求它必须输出固定结构：

```text
主人格、副人格、风险等级、亏钱场景、训练能力、阳明心学提醒、7天行动建议、训练营建议
```

### 4. 报告出现荐股、预测、收益承诺

解决：在系统提示词和用户提示词里都加合规边界：

```text
不荐股，不预测行情，不承诺收益，不给买卖点。
```
