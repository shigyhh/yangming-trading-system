# 九种交易人格测评系统安全方案

当前 `web-mvp` 是本地静态 MVP，登录门用于演示体验和私域承接，不等于真正安全。正式上线时，安全重点不是把前端代码藏起来，而是让核心资产不进入前端。

## 推荐上线架构

浏览器只负责展示题目和提交答案，服务端负责登录、抽题、评分、报告生成和数据留存。

建议链路：

1. 用户进入登录页。
2. 用户通过微信授权、手机号验证码或邀请码登录。
3. 服务端签发 `HttpOnly + Secure + SameSite` Cookie。
4. 前端请求 `/api/assessment/start`。
5. 服务端从题库里抽 27/45/90 道题，只返回本次需要作答的题目。
6. 前端提交答案到 `/api/assessment/submit`。
7. 服务端评分并调用 Coze/OpenAI 生成报告。
8. 服务端把报告和线索同步到飞书、企微或数据库。

## 登录与身份

- 微信登录：优先接公众号 OAuth 或微信开放平台，保存 `openid/unionid`，用于识别同一用户。
- 手机号登录：接短信服务商，验证码 5 分钟过期，单手机号、单 IP 限频。
- 邀请码登录：适合训练营、直播间、内部测试，邀请码要有有效期、使用次数和渠道来源。
- 登录态：不要把长期 token 存在 `localStorage`。正式版使用服务端 Cookie。

## 防爬虫核心策略

- 不在前端暴露完整 3600 题库。前端只拿本次抽到的题。
- 服务端随机抽题，并记录本次 `assessment_id`，答案必须绑定这个 ID。
- 对 `/api/assessment/start` 和 `/api/assessment/submit` 做限流。
- 单 IP、单用户、单设备短时间内频繁开始测评时，触发验证码或人工校验。
- 对异常行为做风控：答题过快、重复提交、连续生成大量报告、同一 IP 多账号。
- 使用 CDN/WAF，例如 Cloudflare、腾讯云 EdgeOne、阿里云 WAF。
- 使用 Turnstile 或滑块验证码，只在异常时触发，不要影响正常用户。
- 服务端校验 `Origin`、`Referer`、CSRF Token、请求时间戳和一次性 nonce。
- 答案提交接口使用幂等键，避免重复刷报告。

## 内容资产保护

- 题库不要放在公开静态目录。
- 报告模板、评分规则、训练营分层逻辑放服务端。
- 每份报告加用户昵称、生成时间、报告 ID 水印，减少被搬运。
- 题目可以设置少量暗标题，用于发现题库泄露来源。
- 后台记录导出、复制、批量访问等高风险行为。

## 数据与合规

- 只收集必要信息：昵称、手机号或微信标识、渠道、测评答案、报告结果。
- 页面明确说明：测评用于交易认知教育和行为训练，不构成投资建议。
- 手机号、微信标识等个人信息要加密存储或至少限制后台访问权限。
- 飞书 Webhook、Coze API Key、短信服务 Key 必须放服务端环境变量，不能写进前端。

## MVP 到正式版的落地顺序

1. 先保留当前登录门，用于直播间和私域体验。
2. 新建服务端接口，把完整题库迁移到服务端。
3. 接微信登录或手机号验证码。
4. 接数据库保存用户、测评、报告。
5. 接限流、验证码、WAF。
6. 再接飞书多维表格和训练营分层后台。
