# 上线发布流程

目标：每次新版本都按同一套动作走，避免“本地正常，服务器却还是旧版”。

## 正常发布

在本地项目根目录执行：

```bash
bash scripts/deploy-mvp.sh
```

脚本会自动完成：

1. 生成带时间版本号的发布包。
2. 给 `styles.css` 和 `app.js` 加版本号，避免浏览器继续缓存旧文件。
3. 上传到服务器。
4. 在服务器备份旧版本。
5. 解压覆盖新版本。
6. 安装后端依赖。
7. 重启服务。
8. 调用 `/health` 做健康检查。

发布完成后，用脚本最后输出的网址验收，例如：

```text
http://116.62.146.166/?v=20260526210000
```

当前服务器建议让 Node 服务跑在 `8787`，公网的 `80` 端口作为访问入口转发到后端。不要随便杀 `80` 端口进程，否则容易出现 502。

## 每次发布前先看三件事

1. 本地页面已经确认没问题。
2. `node --check web-mvp/app.js` 通过。
3. 不要把密码、密钥、`.env` 文件放进发布包。

## 如果页面还是旧的

优先做这三步：

1. 打开脚本输出的带 `?v=` 的新地址。
2. 浏览器强制刷新一次。
3. 看服务器健康检查是否返回 `ok: true`。

## 常见提示

看到下面这类提示，一般不用管：

```text
tar: Ignoring unknown extended header keyword 'LIBARCHIVE.xattr...'
```

这是 Mac 文件扩展属性造成的警告，不代表发布失败。

如果看到：

```text
kill: Operation not permitted
```

说明旧服务没有被正常停止。新版脚本已经改成不会因为这个直接中断；如果页面仍旧没变，再检查服务器日志。

如果网页显示 502，通常是入口还在，但后端服务没有启动。优先在服务器里检查：

```bash
curl http://127.0.0.1:8787/health
```

也可以在本地项目根目录执行只读排查脚本：

```bash
bash scripts/diagnose-server-entry.sh
```

它会检查服务器本机 `8787` 后端、`80` 公网入口、Nginx 转发配置和后端日志，不会修改服务器文件。现在如果 `8787` 返回 `ok: true`，但 `80` 仍然是 502，大概率就是 Nginx 没有转发到 `127.0.0.1:8787`。

如果排查结果显示：`8787` 正常、Nginx 配置里有 `proxy_pass http://127.0.0.1:8787;`，但 `80` 没有监听，就执行：

```bash
bash scripts/fix-server-entry.sh
```

这个脚本会启动 Nginx，并把公网 `80` 入口接到已经运行的后端。

## 如果发布失败

服务器会保留一份备份：

```text
/root/xxjyxt-backup-版本号.tar.gz
```

不要急着乱删文件，先看日志：

```bash
tail -n 80 /root/xxjyxt-server.log
```

## 当前注意

服务器临时密码用完后要尽快更换。后面正式上线域名和备案完成后，建议再加 HTTPS、进程守护、数据库备份和安全限流。
